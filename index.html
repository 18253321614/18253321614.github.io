阶段一：tab coding
辅助编程，代码补全结合智能提示，同时提供一个聊天窗口
代表产品：各种IDE插件，最开始的通义灵码、github Copilot、CodeGeeX

阶段二：Vibe Coding
氛围编程，追求速度和创意
直觉式、对话驱动的编程方式，通常结合cursor等工具。主打跟着感觉走，开发者通过自然语言与AI继续持续对话，快速将模糊的想法转换成可运行的代码。
适合简单的脚本编写、代码补全、快速原型设计。但是一旦项目规模变大，就会遇到代码质量问题，因为AI生成的代码通常不够规范、健壮。
代表产品：cursor、Trae

阶段三：Spec Coding
规范编程，核心是先计划在干活。强调在写代码之前先通过结构化的文档明确需求、设计和技术约束。
先输出产品需求文档，然后进行技术设计，最后列出任务清单（基础构架、数据模型、状态管理、UI组件），这些文档都是spec，
代表产品：Kiro（亚马逊）

在实际开发中，最高效的策略往往是两者的结合，根据项目阶段灵活切换：

  1. 探索阶段 (用 Vibe)：当你不知道需求是否成立，或者想快速验证一个点子时，使用 Vibe Coding 快速“画”出原型14。
  2. 定型阶段 (转 Spec)：当产品方向确定，需要进行工程化落地、团队协作或上线运营时，将核心模块通过 Spec Coding 进行重构和规范化，补齐文档和测试。
  3. 混合模式：即使是使用 Spec Coding 的项目，也可以允许用 Vibe Coding 来探索具体的技术方案或UI细节，验证后再纳入规范体系1。

ReAct运行机制

ReAct是Reasoning + Acting的简写，即推理+行动，字面意思是让LLM一边想（思考和推理）一边做（工具调用）来解决问题。从技术本质上来讲，ReAct不是一种算法，而是LLM+工具集+循环调度机制的集成架构。核心创新是将人类思考问题的认知模式抽象为机器可执行的框架，是机器具备自主拆解任务、动态调整策略的能力

大模型部署

发表于 2026-01-20 分类于 AI 本文字数： 1.8k 阅读时长 ≈ 2 分钟

整体路线

大模型部署首先需要有训练好的模型、推理引擎、服务管理发布平台，常用的推理引擎有vLLM、Hugging Face TGI、Ollama等，其中前两者主要用于企业级模型部署，而Ollama比较适合本地化部署。服务发布管理平台负责管理企业部署的各种模型，例如LLM、Embedding、Rerank等，提供统一的API供其他应用访问，但是Xinference本身也是可以作为推理引擎使用的，并且支持各种模型的推理应用。而Dify在上面的作用是应用开发平台，可以用各种模型的API结合内置的工具快速完成应用构建和开发，和Coze比较相似，只是Coze更适合新手，偏向低代码化。

AutoDL是一个算力市场，可在线租用GPU：https://www.autodl.com/home

主线1

对于企业来说，数据安全至关重要，这也是企业私有化部署大模型服务的主要原因，但是除非企业有自己的机房和私有云系统，最好的方式还是采用云服务厂商的云服务，数据安全方面也有云服务厂商做保障，购买后的与资源也是私有的不用担心泄露。

我们可以使用腾讯云部署Dify的docker容器，首先购买服务器，然后在服务器上安装dokcer，使用腾讯云的镜像加速服务中的镜像源地址，使用docker拉取Dify即可完成部署，默认使用80端口，直接访问即可。

这样Dify服务就快速部署好了。接下来就是在Dify中设置大模型的服务，既可以使用各大厂商提供的API服务，也可以使用在自己服务器上部署的模型服务。

主线2

注意：AutoDL可以开启学术加速，下载国外资源失败时可以

主线2使用一句话概括，就是在AutoDL的服务器上安装Ollama后，使用Ollama命令一键部署大模型服务。

在AutoDL中租赁好服务器后，使用脚本安装Ollama，安装好启动Ollama服务，例如OLLAMA_HOST=27.0.0.1:6006 ollama serve（这里指定了开放端口，AutoDL只开放6006）。接着就可以使用Ollama拉取模型，模型名称可以在Ollama的市场中寻找，模型部署好，我们就完成了大模型服务API的部署，接着在Dify就可以调用AutoDL中部署的Ollama服务了。

主线3

在主线三中Xinfraence作为一个推理框架使用，首先在AutoDL市场中租赁服务器和显卡资源，使用pip命令可以直接安装Xinference后启动服务并暴露端口，然后在Xinference中下载模型即可。最后配置Dify和Xinference连接。

补充

  1. 推理服务引擎（如 vLLM、Ollama）在大模型应用中的作用，可以类比为“餐厅的后厨引擎”或“汽车的发动机”。它的核心使命不是让你“拥有模型”（那是下载器的事），也不是让你“管理模型”（那是管理平台的事），而是如何在有限的硬件资源下，以最快的速度、最低的成本，处理最多的用户请求。

  2. 目前推理引擎用的比较多还有vLLM，如果你正在搭建一个高并发的API服务，追求每秒处理最多的Token，且只关注文本生成，选 vLLM，它是性能的首选，虽然文档未直接列出，但它是构建高性能服务的基石，Xinference和Ollama使用资源较大。

  3. 大语言模型推理（LLM Inference） ：是指利用已经训练好的模型参数，将用户输入的文本（Prompt）转化为模型输出的文本（Response）的计算过程。

  4. Ollama不支持重排序模型

问题

上面集中方式都是直接下载大模型，如果自己使用预训练模型微调了一个模型，该怎么部署

这里以LoRA微调为例，微调完成后会产生一个文件夹，里面包含adapter_config.json 和 adapter_model.bin 等文件，在Ollama中可以直接加载，但是要告诉Ollama使用的基座模型是什么。

如果使用的是全量微调并且对模型架构进行了修改之后在微调得到的模型会复杂一些，但本质都是两个东西，模型架构和权重参数，如果仅仅是修改权重参数，模型架构保持不变，那么各种推理框架都可以快速部署。但是入股修改了模型架构，就需要让推理框架了解你的模型架构。
