ReAct Agent

发表于 2026-03-06 分类于 AI 本文字数： 10k 阅读时长 ≈ 9 分钟

ReAct运行机制

ReAct是Reasoning + Acting的简写，即推理+行动，字面意思是让LLM一边想（思考和推理）一边做（工具调用）来解决问题。从技术本质上来讲，ReAct不是一种算法，而是LLM+工具集+循环调度机制的集成架构。核心创新是将人类思考问题的认知模式抽象为机器可执行的框架，是机器具备自主拆解任务、动态调整策略的能力

核心机制：T-A-O循环

Thought（思考） –> Action（行动） –> Observation （观察） –> Thought（思考），循环下去

  • Thought：模型的内心独白，思考和推理的过程
  • Acting：工具调用
  • Observation：外部环境对Act的客观反馈，如web搜索结果、文件控制等

例如：当人们计划一次旅行时，首先会思考需求（Thought），找地点、看交通、订酒店（Acting），找到这些之后，再修改计划（Observation），形成循环。

与思维链和思维树相比：

  • CoT：思维链，内部推理完全依赖模型内部的权重，不会多次调用模型。
  • ToT：思维树，思考与回溯，每一步都要思考有几种可能，然后执行，推理成本很高。
  • ReAct：多轮调用，每一轮都可以使用外部工具，并对每轮的回复进行分析（环境反馈闭环）

CoT只有脑子，ToT脑子想的太复杂，ReAct既有脑子又有手，而且能够通过外部反馈实时调整策略。

与if-else相比：有很强的灵活性和容错率，交给LLM自助决策，if-else智能处理有限的情形；高度可解释性，会将每一步的T-O-A显示出来。

ReAct五大构成要素：

  1. 历史上下文：Agent记忆，维护一个历史对话，记录之前所有的思考、行动、观察
  2. 环境信息：Agent的感知，在当前时刻接受到的外部信息（如用户原始输入、系统状态），作为推理的输入条件，帮助决定下一步
  3. LLM：Agent的大脑，负责推理和决策。每次需要思考时，LLM会结合记忆和感知，生成行动计划。
  4. 工具/动作：Agent的手脚
  5. 观察结果：Agent的反馈，每次工作返回的结果。包括成功的数据、错误日志等，这个反馈将进入下一步推理的输入

案例推演

Prompt构造，提示词是核心，需包含【角色定义、可用工具、思考行动规则、示例、历史上下文与当前问题】

案例推演：请帮我查询一下，北京今天天气怎么样，适合穿什么样的衣服，并保存下来

轮次 (Round) 历史记忆 (History) 当前轮次思考 (Thought) 行动 (Action / ToolCall) 观察结果 (Observation)
Round 1 空 “我需要知道北京今天的天气，需要调用天气工具。” 调用工具: 天气查询 参数: “北京” 返回: 温度25℃, 晴天, 湿度60%, 风速3m/s
Round 2 已获取北京天气数据 “现在我知道天气了，需要查询知识库获取具体的穿衣建议。” 调用工具: 穿衣知识库 参数: “25度晴天” 返回: 建议穿短袖T恤, 带薄外套防温差, 建议防晒
Round 3 天气数据(25℃); 已获取穿衣建议 “我已经获取了完整信息，现在需要将这些建议写入文件中保存。” 调用工具: writeToFile 参数: 组合好的文本 返回: 建议已写入文件 weather.md
Round 4 天气数据; 穿衣建议; 已保存文件 “文件已保存，用户的目标已经全部达成，我可以输出最终回复了。” 直接输出最终答案 任务圆满结束!

ReAct流程

总结案例的过程，可以分为三步：初始化——循环迭代（TAO）——终止输出。

初始化阶段

  1. 任务解析，明确任务的目标、约束和任务类型
  2. 初始化上下文管理，保存迭代揭露
  3. 提供FewShot案例，即任务-思考-行动-观察-结果的链路

循环迭代阶段

进行TAO循环，并保存到上下文管理器中，形成可追溯的推理过程。

  1. Thought，决策逻辑生成：核心输出是当前任务的进展和下一步计划

    示例推理：“当前任务是查明天从深圳到海南的航班，要选最便宜且晚上的并预订，历史未获取任何数据。需先调用航班查询工具，参数包含出发地深圳、目的地海南、日期明天、时段晚上，获取符合条件的航班列表后再筛选最便宜的，第一步调用航班查询工具。”

  2. Act，标准化执行指令输出：推理结果转化为标准的行动指令，必须包含工具名称和参数

行动类型 核心功能 格式示例
信息检索类 获取外部事实性信息 flight_search[深圳,海南,明天,晚上]
数据处理类 筛选、排序等数据处理 filter_sort[航班列表,价格,升序]
服务预订类 预订机票、酒店等服务 flight_book[CA1234,张三,身份证123456789012345678]
结果输出类 提交最终任务结果 finish[明天深圳到海南最便宜晚上航班为CA1234（20:00-21:30），票价500元，已完成预订，乘客张三]

  1. Observation，客观反馈获取：获取工具调用的结果

    示例：航班查询行动“flight_search[深圳,海南,明天,晚上]”的观察结果为“航班列表：1.CA1234（20:00-21:30，票价500元）；2.CZ5678（21:10-22:40，票价650元）；3.MU9012（19:30-21:00，票价580元）”；若查询失败，观察结果为“航班查询工具调用超时，未获取航班数据”。

终止输出阶段

  1. 终止条件判断：Finish任务完成、最大迭代次数、3次失败（熔断机制）
  2. 结果整理并生成答案

代码实战

这是一个天气查询和计算的案例，使用了langchian1.0实现，但是手动实现了核心的TAO逻辑。首先定义了模型调用的三个参，然后初始化了OpenAI标准的LLM，接着定义了两个工具，也就是两个函数，并放到了注册表中（一个字典）。

ManualReActAgent中的ReAct行为的实现，主要包含四个方法：_get_react_prompt_template _parse_action _run_tool 和invoke

  • _get_react_prompt_template：定义了提示词模板，包括输出格式、上下文、工具等
  • _parse_action：用来解析模型的输出，判断是否需要进行工具调用，如果返回中包含final则说明不需要工具调用
  • _run_tool ：如果_parse_actio返回了tool_call，则执行该方法，返回调用结果
  • invoke：上述三个方法不断迭代，每次迭代都将TAO放到history中，放到下一轮循环的提示词中，直到循环结束或者模型输出final_answer、或者报错，调用才算结束。

阶段一：tab coding
辅助编程，代码补全结合智能提示，同时提供一个聊天窗口
代表产品：各种IDE插件，最开始的通义灵码、github Copilot、CodeGeeX

阶段二：Vibe Coding
氛围编程，追求速度和创意
直觉式、对话驱动的编程方式，通常结合cursor等工具。主打跟着感觉走，开发者通过自然语言与AI继续持续对话，快速将模糊的想法转换成可运行的代码。
适合简单的脚本编写、代码补全、快速原型设计。但是一旦项目规模变大，就会遇到代码质量问题，因为AI生成的代码通常不够规范、健壮。
代表产品：cursor、Trae

阶段三：Spec Coding
规范编程，核心是先计划在干活。强调在写代码之前先通过结构化的文档明确需求、设计和技术约束。
先输出产品需求文档，然后进行技术设计，最后列出任务清单（基础构架、数据模型、状态管理、UI组件），这些文档都是spec，
代表产品：Kiro（亚马逊）

在实际开发中，最高效的策略往往是两者的结合，根据项目阶段灵活切换：

  1. 探索阶段 (用 Vibe)：当你不知道需求是否成立，或者想快速验证一个点子时，使用 Vibe Coding 快速“画”出原型14。
  2. 定型阶段 (转 Spec)：当产品方向确定，需要进行工程化落地、团队协作或上线运营时，将核心模块通过 Spec Coding 进行重构和规范化，补齐文档和测试。
  3. 混合模式：即使是使用 Spec Coding 的项目，也可以允许用 Vibe Coding 来探索具体的技术方案或UI细节，验证后再纳入规范体系1。

大模型部署

发表于 2026-01-20 分类于 AI 本文字数： 1.8k 阅读时长 ≈ 2 分钟

整体路线

大模型部署首先需要有训练好的模型、推理引擎、服务管理发布平台，常用的推理引擎有vLLM、Hugging Face TGI、Ollama等，其中前两者主要用于企业级模型部署，而Ollama比较适合本地化部署。服务发布管理平台负责管理企业部署的各种模型，例如LLM、Embedding、Rerank等，提供统一的API供其他应用访问，但是Xinference本身也是可以作为推理引擎使用的，并且支持各种模型的推理应用。而Dify在上面的作用是应用开发平台，可以用各种模型的API结合内置的工具快速完成应用构建和开发，和Coze比较相似，只是Coze更适合新手，偏向低代码化。

AutoDL是一个算力市场，可在线租用GPU：https://www.autodl.com/home

主线1

对于企业来说，数据安全至关重要，这也是企业私有化部署大模型服务的主要原因，但是除非企业有自己的机房和私有云系统，最好的方式还是采用云服务厂商的云服务，数据安全方面也有云服务厂商做保障，购买后的与资源也是私有的不用担心泄露。

我们可以使用腾讯云部署Dify的docker容器，首先购买服务器，然后在服务器上安装dokcer，使用腾讯云的镜像加速服务中的镜像源地址，使用docker拉取Dify即可完成部署，默认使用80端口，直接访问即可。

这样Dify服务就快速部署好了。接下来就是在Dify中设置大模型的服务，既可以使用各大厂商提供的API服务，也可以使用在自己服务器上部署的模型服务。

主线2

注意：AutoDL可以开启学术加速，下载国外资源失败时可以

主线2使用一句话概括，就是在AutoDL的服务器上安装Ollama后，使用Ollama命令一键部署大模型服务。

在AutoDL中租赁好服务器后，使用脚本安装Ollama，安装好启动Ollama服务，例如OLLAMA_HOST=27.0.0.1:6006 ollama serve（这里指定了开放端口，AutoDL只开放6006）。接着就可以使用Ollama拉取模型，模型名称可以在Ollama的市场中寻找，模型部署好，我们就完成了大模型服务API的部署，接着在Dify就可以调用AutoDL中部署的Ollama服务了。

主线3

在主线三中Xinfraence作为一个推理框架使用，首先在AutoDL市场中租赁服务器和显卡资源，使用pip命令可以直接安装Xinference后启动服务并暴露端口，然后在Xinference中下载模型即可。最后配置Dify和Xinference连接。

补充

  1. 推理服务引擎（如 vLLM、Ollama）在大模型应用中的作用，可以类比为“餐厅的后厨引擎”或“汽车的发动机”。它的核心使命不是让你“拥有模型”（那是下载器的事），也不是让你“管理模型”（那是管理平台的事），而是如何在有限的硬件资源下，以最快的速度、最低的成本，处理最多的用户请求。

  2. 目前推理引擎用的比较多还有vLLM，如果你正在搭建一个高并发的API服务，追求每秒处理最多的Token，且只关注文本生成，选 vLLM，它是性能的首选，虽然文档未直接列出，但它是构建高性能服务的基石，Xinference和Ollama使用资源较大。

  3. 大语言模型推理（LLM Inference） ：是指利用已经训练好的模型参数，将用户输入的文本（Prompt）转化为模型输出的文本（Response）的计算过程。

  4. Ollama不支持重排序模型

问题

上面集中方式都是直接下载大模型，如果自己使用预训练模型微调了一个模型，该怎么部署

这里以LoRA微调为例，微调完成后会产生一个文件夹，里面包含adapter_config.json 和 adapter_model.bin 等文件，在Ollama中可以直接加载，但是要告诉Ollama使用的基座模型是什么。

如果使用的是全量微调并且对模型架构进行了修改之后在微调得到的模型会复杂一些，但本质都是两个东西，模型架构和权重参数，如果仅仅是修改权重参数，模型架构保持不变，那么各种推理框架都可以快速部署。但是入股修改了模型架构，就需要让推理框架了解你的模型架构。
