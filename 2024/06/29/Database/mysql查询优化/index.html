mysql查询优化

发表于 2024-06-29 分类于 数据库 本文字数： 11k 阅读时长 ≈ 10 分钟

性能下降的原因

  1. 查询语句写的烂
  2. 未使用索引或者索引失效
  3. 关联查询太多
  4. 服务器问题，例如服务器参数设置、线程池、缓冲区等的问题

🏅补充：

MySQL的数据是存储在磁盘文件中的，查询处理数据时，需要先把磁盘中的数据加载到内存中，磁盘IO 操作非常耗时，所以我们优化的重点就是尽量减少磁盘 IO 操作。访问二叉树的每个节点就会发生一次IO，如果想要减少磁盘IO操作，就需要尽量降低树的高度。那如何降低树的高度呢？

假如key为bigint=8字节，每个节点有两个指针，每个指针为4个字节，一个节点占用的空间16个字节（8+42=16）。

因为在MySQL的InnoDB存储引擎一次IO会读取的一页（默认一页16K）的数据量，而二叉树一次IO有效数据量只有16字节，空间利用率极低。为了最大化利用一次IO空间，一个简单的想法是在每个节点存储多个元素，在每个节点尽可能多的存储数据。每个节点可以存储1000个索引（16k/16=1000），这样就将二叉树改造成了多叉树，通过增加树的叉树，将树从高瘦变为矮胖。构建1百万条数据，树的高度只需要2层就可以（10001000=1百万），也就是说只需要2次磁盘IO就可以查询到数据。磁盘IO次数变少了，查询数据的效率也就提高了。

哪些情况需要创建索引:

  1. 主键自动创建唯一索引

  2. 频繁作为查询条件的字段应当建立索引

  3. 查询中与其他表关联的字段，外间关系建立索引

  4. 频繁更新的字段不适合创建索引。

  5. where条件里用不到的字段不创建索引

  6. 单键/组合索引的选择问题（高并发下倾向创建组合索引）

  7. 查询中的排序字段，排序字段若通过索引去访问将大大提高排序速度。

  8. 查询中统计或者分组字段

哪些情况不要创建索引:

  1. 标记路太少
  2. 经常增删改的表
  3. 字段重复率很高的的字段，对字段加索引没什么效果。
  4. 索引选择性：索引列中不同值的数量和总记录数的比值，例如一个表中有2000条数据，索引列有1980个不同的值，这时候索引选择性=0.99，这时候索引的效率就很高。

Select语句的执行过程

基本执行过程

  1. 解析查询：MySQL首先解析SQL查询，检查语法是否正确，确定查询涉及的表和列。
  2. 查询优化：查询优化器会分析查询，选择最有效的执行计划。这可能涉及决定使用哪个索引，以及在多表查询中的表的连接顺序。
  3. 读取数据：根据优化器的决定，MySQL会从磁盘读取必要的数据页到内存中。如果数据页已经在内存中（例如，在缓冲池中），则直接使用。
    • 如果查询使用了索引，MySQL会首先读取索引，然后根据索引找到对应的数据页。
      • 如果没有使用索引，MySQL会进行全表扫描，读取所有的数据页。
  1. 执行查询：在内存中，MySQL执行查询，过滤和处理数据。对于SELECT查询，这可能包括应用WHERE子句的条件，执行连接，分组和排序数据，以及计算聚合函数。
  2. 返回结果：最后，MySQL将结果集返回给客户端。结果集是满足查询条件的行的集合。

示例

以InnoDB为例：

  1. 读取索引表到内存
  2. 索引查找：MySQL首先在索引表中查找满足username='great'的索引项。这个过程通常比全表扫描要快，因为索引结构允许MySQL快速定位到满足条件的索引项（B+树种搜索）。
  3. 行查找（回表）：对于每个满足条件的索引项，MySQL会使用索引项中的行指针（对于非聚簇索引）或主键值（对于聚簇索引）来查找对应的数据行。这个过程可能涉及从磁盘读取数据页到内存。
  4. 行过滤：MySQL在内存中对每个找到的数据行应用age > 18条件进行过滤。只有满足条件的行才会被包含在结果集中。
  5. 结果返回：最后，MySQL将结果集返回给客户端。

请注意，这个过程可能会因为查询的具体内容和数据库的配置而有所不同。例如，如果查询只需要索引中的信息（即，覆盖索引查询），那么MySQL可能不需要查找和过滤数据行。

索引优化分析

mysql服务器中，优化器负责对用户的query请求进行优化。当客户端向MySQL请求一条Query，解析器对请求进行分类，将select请求转发给优化器模块，根据query进行一些分析计算，然后得出最后的执行计划。

优化重点

MySQL常见瓶颈：

  • CPU利用率高，一般发生在数据装入内存或从磁盘上读取数据的时候
  • 一次查询IO次数过多，装入数据远大于内存容量的时候(不停的交换)
  • 服务器性能瓶颈

因此对于查询的优化重点就是降低CPU利用率和I/O次数

Explain执行计划分析

使用explain关键字可以模拟优化器执行sql执行语句，从而知道mysql是如何处理你的mysql语句。进而可以分析sql执行的性能瓶颈。

‼️注意：

一个负责SQL语句通常会经过多次查询，最终将数据组合得到结果。Explain执行计划其实就是将一个SQL语句的多次查询的详情列出来。

使用方式：explain + SQL语句

查询计划各字段的含义

Id select_type table type Possible_keys key Key_len Ref extra

id

用来表示查询某个表的执行顺序，id越大执行优先级越高，id相同的查询执行顺序从上到下依次执行

select_type

常见类型如下：

SIMPLE 表示简单的查询
PRIMARY 查询中包含任何复杂的子查询或者UNION
DERIVED 在from中出现的子查询会被标注为DERIVED，MySQL会递归执行这些子查询，并将结果放在临时表中
SUBQUERY 在select或者where中包含的子查询会被标注为SUBQUERY

table

查询的表，注意上图中<derived2>表示id为2的select_type为DERIVED的查询结果临时表

type

查询效率从上到下一次降低，ref级别就可以接受

system 表里只有一行记录（等于系统表），属于const的特例，平时几乎不会出现可以忽略
const 索引一次就找到了，const用于比较primary key或者unique索引，只匹配一行所以很快。const的意思是使用唯一索引等于一个常量的查询。（例如：将主键置于where列表中，MySQL就可以把查询转换为一个常量）
eq_ref 唯一性索引扫描 ，对于每个索引键，表中只有一条记录与之匹配。常见于主键和唯一索引扫描。
ref 非唯一索引扫描，用到了索引，但是索引不是唯一索引，是普通的可重复的索引。
range 使用索引进行一个范围查询
index Full Index Scan，index与all的区别为index类型只遍历索引树（索引单独存储在一张表中）。因为索引表通常比数据文件小，所以index效率要比all效率高
ALL 将遍历全表以找到满足条件的行（在全表扫描过程中，MySQL会将表中的数据分页读取到内存中，然后对每一行应用WHERE子句中的条件进行筛选过滤。满足条件的行会被包含在结果集中。）

执行过程分析：

根据explain分析结果，首先读取t2表中数据，采用全索引扫描的方式，拿到t2所有数据后。之后有两种选择：

  • 情形1：取出t1所有数据，根据on条件进行连接
  • 情形2：根据on条件取出t1中符合条件的数据，最后进行连接

根据explain结果可以看到使用了eq_ref唯一性索引扫描，也就是在执行t1的查询时，使用具有唯一索引的id作为查询条件，也就是先执行的ON。具体过程是：读取t1表之前先检查是否有索引，将索引读入内存，然后根据索引找到对应的数据页。

possible keys

显示可能应用在这张表的索引，一个或多。查询涉及到的字段如果存在索引，则该索引会被列出来，但不一定会实际使用。

key

查询中实际使用的索引，如果时null，则没有使用索引

key_length

越大，索引使用越充分

ref

显示索引的哪一列被使用了

rows

显示MySQL认为它执行查询时必须检查的行数。

extra

using filesot mysql无法利用索引完成排序
using temporary mysql对查询结果进行排序时使用的了临时表，常见于oeder by和group by查询（分组没有使用索引）
using index 表示索引被用来执行索引键值的查找，避免访问表数据行，效率还不错（仅仅从索引种就查询出了需要的数据，没有回表）
using where 使用了where过滤条件

索引失效的情况

index(a,b,c)

全值匹配我最爱，最左前缀要遵守； where a=3 and c=5 可以用到a
带头大哥不能死，中间兄弟不能断； where a=3 and c=5 and b=4 a\b\c，会自动调整顺寻
索引列上少计算，范围之后全失效；
Like百分写最右，覆盖索引不写星； where a=3 and b like ‘kk%’ and c=4 可以用到a\b\c
不等空值还有or，索引失效要少用；
VAR引号不可丢，SQL高级也不难！

🏅建议：

  1. 对于单值索引，选择针对当前查询字段过滤性更好的索引
  2. 复合索引将过滤性更好的字段放在前面
  3. 组合索引，尽量包含where子句种更多的字段
  4. 通过分析统计信息和调整query的写法达到选择合适索引的目的

排序优化

避免filesort

order by语句使用索引最左前列活使用where子句与order by子句条件满足最左前列。

例：KEY a_b_c(a,b,c)

‼️‼️‼️‼️‼️注意：

where a=const order by b这种情况，where a=const g > const order by b是不影响复合索引a_b_c生效的。就是说乱入不会影响原有的复合索引。

补充：

  1. filesort的两种排序算法
  • 双路排序：是首先根据相应的条件取出相应的排序字段和可以直接定位行数据的行指针信息，然后在sort buffer 中进行排序。排序后再吧查询字段依照行指针取出，共执行两次磁盘io。
  • 单路算法：是一次性取出满足条件行的所有字段，然后在sort buffer中进行排序。 执行一次磁盘io。（sortbuffer要足够大【通过sort_buffer_size参数设置】，否则还是会变成双路排序）
  1. Mysql如何选择

通过比较系统参数 max_length_for_sort_data的大小和Query 语句所取出的字段类型大小总和来判定需要使用哪一种排序算法。

  • max_length_for_sort_data大于数据总和，使用单路
  • max_length_for_sort_data小，使用多路

关联查询

  • 关联查询，左表会使用全表扫描（所以左表关联字段加索引没有意义），右表的关联字段应该添加索引。
  • 左表作为驱动表，应该使用数据量少的表

注意：

  • 尽量减少join语句中嵌套循环的总次数（join (select * from a join b on~~)），永远用小的结果集驱动大的结果集
  • 优先优化嵌套循环的内部循环
  • 保证被驱动表的join条件字段已被索引
  • 当无法保证被驱动表的join条件字段被索引并且内存充足的条件下，不要吝啬joinbuffer
