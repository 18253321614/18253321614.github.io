ReAct Agent

ReAct运行机制

ReAct是Reasoning + Acting的简写，即推理+行动，字面意思是让LLM一边想（思考和推理）一边做（工具调用）来解决问题。从技术本质上来讲，ReAct不是一种算法，而是LLM+工具集+循环调度机制的集成架构。核心创新是将人类思考问题的认知模式抽象为机器可执行的框架，是机器具备自主拆解任务、动态调整策略的能力

核心机制：T-A-O循环

Thought（思考） –> Action（行动） –> Observation （观察） –> Thought（思考），循环下去

  • Thought：模型的内心独白，思考和推理的过程
  • Acting：工具调用
  • Observation：外部环境对Act的客观反馈，如web搜索结果、文件控制等

例如：当人们计划一次旅行时，首先会思考需求（Thought），找地点、看交通、订酒店（Acting），找到这些之后，再修改计划（Observation），形成循环。

与思维链和思维树相比：

  • CoT：思维链，内部推理完全依赖模型内部的权重，不会多次调用模型。
  • ToT：思维树，思考与回溯，每一步都要思考有几种可能，然后执行，推理成本很高。
  • ReAct：多轮调用，每一轮都可以使用外部工具，并对每轮的回复进行分析（环境反馈闭环）

CoT只有脑子，ToT脑子想的太复杂，ReAct既有脑子又有手，而且能够通过外部反馈实时调整策略。

与if-else相比：有很强的灵活性和容错率，交给LLM自助决策，if-else智能处理有限的情形；高度可解释性，会将每一步的T-O-A显示出来。

ReAct五大构成要素：

  1. 历史上下文：Agent记忆，维护一个历史对话，记录之前所有的思考、行动、观察
  2. 环境信息：Agent的感知，在当前时刻接受到的外部信息（如用户原始输入、系统状态），作为推理的输入条件，帮助决定下一步
  3. LLM：Agent的大脑，负责推理和决策。每次需要思考时，LLM会结合记忆和感知，生成行动计划。
  4. 工具/动作：Agent的手脚
  5. 观察结果：Agent的反馈，每次工作返回的结果。包括成功的数据、错误日志等，这个反馈将进入下一步推理的输入

案例推演

Prompt构造，提示词是核心，需包含【角色定义、可用工具、思考行动规则、示例、历史上下文与当前问题】

案例推演：请帮我查询一下，北京今天天气怎么样，适合穿什么样的衣服，并保存下来

轮次 (Round) 历史记忆 (History) 当前轮次思考 (Thought) 行动 (Action / ToolCall) 观察结果 (Observation)
Round 1 空 “我需要知道北京今天的天气，需要调用天气工具。” 调用工具: 天气查询 参数: “北京” 返回: 温度25℃, 晴天, 湿度60%, 风速3m/s
Round 2 已获取北京天气数据 “现在我知道天气了，需要查询知识库获取具体的穿衣建议。” 调用工具: 穿衣知识库 参数: “25度晴天” 返回: 建议穿短袖T恤, 带薄外套防温差, 建议防晒
Round 3 天气数据(25℃); 已获取穿衣建议 “我已经获取了完整信息，现在需要将这些建议写入文件中保存。” 调用工具: writeToFile 参数: 组合好的文本 返回: 建议已写入文件 weather.md
Round 4 天气数据; 穿衣建议; 已保存文件 “文件已保存，用户的目标已经全部达成，我可以输出最终回复了。” 直接输出最终答案 任务圆满结束!

ReAct流程

总结案例的过程，可以分为三步：初始化——循环迭代（TAO）——终止输出。

初始化阶段

  1. 任务解析，明确任务的目标、约束和任务类型
  2. 初始化上下文管理，保存迭代揭露
  3. 提供FewShot案例，即任务-思考-行动-观察-结果的链路

循环迭代阶段

进行TAO循环，并保存到上下文管理器中，形成可追溯的推理过程。

  1. Thought，决策逻辑生成：核心输出是当前任务的进展和下一步计划

    示例推理：“当前任务是查明天从深圳到海南的航班，要选最便宜且晚上的并预订，历史未获取任何数据。需先调用航班查询工具，参数包含出发地深圳、目的地海南、日期明天、时段晚上，获取符合条件的航班列表后再筛选最便宜的，第一步调用航班查询工具。”

  2. Act，标准化执行指令输出：推理结果转化为标准的行动指令，必须包含工具名称和参数

行动类型 核心功能 格式示例
信息检索类 获取外部事实性信息 flight_search[深圳,海南,明天,晚上]
数据处理类 筛选、排序等数据处理 filter_sort[航班列表,价格,升序]
服务预订类 预订机票、酒店等服务 flight_book[CA1234,张三,身份证123456789012345678]
结果输出类 提交最终任务结果 finish[明天深圳到海南最便宜晚上航班为CA1234（20:00-21:30），票价500元，已完成预订，乘客张三]

  1. Observation，客观反馈获取：获取工具调用的结果

    示例：航班查询行动“flight_search[深圳,海南,明天,晚上]”的观察结果为“航班列表：1.CA1234（20:00-21:30，票价500元）；2.CZ5678（21:10-22:40，票价650元）；3.MU9012（19:30-21:00，票价580元）”；若查询失败，观察结果为“航班查询工具调用超时，未获取航班数据”。

终止输出阶段

  1. 终止条件判断：Finish任务完成、最大迭代次数、3次失败（熔断机制）
  2. 结果整理并生成答案

代码实战

这是一个天气查询和计算的案例，使用了langchian1.0实现，但是手动实现了核心的TAO逻辑。首先定义了模型调用的三个参，然后初始化了OpenAI标准的LLM，接着定义了两个工具，也就是两个函数，并放到了注册表中（一个字典）。

ManualReActAgent中的ReAct行为的实现，主要包含四个方法：_get_react_prompt_template _parse_action _run_tool 和invoke

  • _get_react_prompt_template：定义了提示词模板，包括输出格式、上下文、工具等
  • _parse_action：用来解析模型的输出，判断是否需要进行工具调用，如果返回中包含final则说明不需要工具调用
  • _run_tool ：如果_parse_actio返回了tool_call，则执行该方法，返回调用结果
  • invoke：上述三个方法不断迭代，每次迭代都将TAO放到history中，放到下一轮循环的提示词中，直到循环结束或者模型输出final_answer、或者报错，调用才算结束。

扩展与未来发展

扩展

分类 内容 补充说明
核心定义 ReAct 范式通过将思考和行动解耦，并循环迭代，使得 LLM 真正具备了与现实世界打交道的能力。 全称：Reason + Act（推理 + 行动）核心逻辑：思考→行动→观察→迭代推理，形成闭环
核心应用场景 需要实时信息的任务 如：实时问答、新闻查询、天气 / 股票查询、实时数据获取
依赖私有知识库的任务 如：企业文档问答、内部资料检索、RAG 结合私有数据问答
多步推理的任务 如：数学解题、逻辑推理、复杂决策、多条件问题求解
与物理 / 数字环境交互的任务 如：工具调用、API 操作、机器人控制、网页交互、软件操作
功能扩展与进阶玩法 ReAct 与自我反思 行动后复盘推理过程，修正错误、优化思路，提升准确性与鲁棒性
ReAct 与全局规划 先做顶层任务拆解，再分步执行，解决复杂长链任务
动态工具检索 根据任务需求自动选择 / 调用工具，适配多样化场景

未来发展

  • 从API调用走向“系统级控制“，例如豆包手机那样

  • 多模态TOA：现在还仅仅是文字

  • 多智能体协同：subagent协作，Agent之间上下文隔离

  • 端侧化与极速推理：无需网络，低成本
