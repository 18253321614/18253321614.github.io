产品设计

当我们构建了产品需求池之后，接下来就是产品设计，产品设计首先要有设计理念，其次是遵循一定的结构。

产品设计理念

产品和产品设计

什么是产品：产品是用来解决某个问题东西，可以是有形的也可以是无形的

什么是产品设计：互联网产品设计是指基于战略需要和用户需要，形成方案，直至方案完成为止的一系列工作的准备和管理。

产品架构设计：产品设计的最顶层，描述整个产品运作的架构原理，告诉我们各个单元之间怎么运作。是技术和业务的一个融合

产品设计流程

产品设计方法

人性法：任性、恐惧、懒惰、虚荣、贪婪、从众

映射法：线上线下映射

参考法：另一个名称叫抄袭，也需要微创新

产品结构设计

结构设计是原型和PRD的基础，即是以什么样的结构组织产品的功能，属于产品设计流程的结构层中的内容。

什么是产品结构

产品结构是基于前期的需求分析和市场及竞品分析，将各个需求点以某种逻辑，系统化的组织起来所形成的立体的结构。

常见的产品结构

线性：具有明显的先后顺序，例如DD打车，就是一步步的跟着流程走，每走一步不需要用户选择。还有**税务申报软件 (如 TurboTax)**：为了确保数据逻辑正确，强制用户按步骤填写个人信息、收入、扣除项等，不能跳跃

层级：信息从概括到具体，从一般到特殊进行组织。用户从主页（根节点）进入大类，再进入子类，最后到达具体内容页。这是互联网最古老也最常见的结构。典型的就是操作系统的文件管理器：磁盘 -> 文件夹 -> 子文件夹 -> 文件。

矩阵：允许用户通过多种不同的维度（分类法）来浏览同一组内容。用户可以根据自己的需求选择“怎么找”，而不是被迫接受唯一的分类路径，一个功能可以被其他功能调用。例如电商详情页，其中包含产品展示（信息展示、评价、包装规则、售后）、商品推荐（看了又看、达人推荐、键盘热搜榜、广告位）、购买入口（购买当前商品、购买套餐、购买同类物品）、进入商户（店内分类、店内搜索）

自然结构：没有结构，例如抖音，鼓励人们探索

结构类型 核心逻辑 用户控制力 典型隐喻 最佳适用
线性 顺序执行 低 (只能前后) 电影胶片 任务流程、教学
层级 父子包含 中 (需回溯) 图书馆书架 内容归档、电商目录
矩阵 多维交叉 高 (自由筛选) 超市货架 + 筛选器 复杂数据检索、媒体库
自然 关联跳转 极高 (自由漫游) 大脑神经元 / 蜘蛛网 知识探索、创意灵感

产品结构分析案例：QQ，分析角度：用户特性、战略方向、产品定位、功能相关性

战略定位：年轻人的社交工具

用户特性：年轻人、查看动态、生活和学习沟通

常见的构建方法

产品的主要功能架构是骨骼，具体功能是血肉，设计时先骨骼还是先血肉：

  • 从下往上：思考每个场景包含哪些功能、有哪些页面，这样做的好处是不会忽略功能细节，缺点是会导致功能设计变得复杂。同时有一些东西只有从整体考虑时才会意识到。

  • 从上往下：从上到下按模块划分，可以更好关注各模块间的关系

在实践中，我们需要从两个角度分别考虑。目前大多数产品采用的都是层级结构，尤其是手机应用，对于PC显示面积大，可以采用矩阵结构。

具体来说，构建产品结构（Information Architecture & Product Structure）不仅仅是画流程图，而是一个从理解用户到定义逻辑，再到可视化呈现的系统工程。

需求串联

首先采用用户旅程映射或故事地图将所有需求串联起来

方法名称 原理 操作步骤 适用场景
用户旅程映射(User Journey Mapping) 不直接设计功能，而是先还原用户完成某个目标的全过程（从产生需求到解决问题）。 1. 列出阶段：如 发现 → 考虑 → 购买 → 使用 → 复购。 2. 识别要素：分析每个阶段的痛点、情绪曲线和关键任务。 3. 转化设计：将“关键任务”转化为核心模块，将“痛点”转化为需优化的功能点。 • 新产品从0到1构建 • 旧产品流程重构
故事地图(User Story Mapping) 将用户活动按时间顺序（横向）和优先级（纵向）排列。 1. 骨干 (Backbone)：横向列出用户主流程（构建线性结构基础）。 2. 任务 (Tasks)：在主流程下挂具体的子任务。 3. 切片 (Slicing)：纵向切割，第一行定义为 MVP，下方为后续迭代版本。 • 敏捷开发中的版本规划 • 平衡线性流程与功能丰富度

结构设计

确定好内容后，如何将其组织成具体的结构形态（线性/层级/矩阵/自然）。

方法名称 原理 操作步骤 适用场景
站点地图法 (Site Mapping) 传统的树状图绘制，明确父子关系。 1. 从根开始：从首页（Root）出发。 2. 层级拆解：向下拆解一级、二级、三级页面。 3. 标注类型：标记页面属性（列表页、详情页、表单页等）。 4. 控制尺度：深度通常不超过3-4层；宽度每层选项控制在7±2个（符合米勒定律）。 • 构建层级结构 • 梳理全站导航体系
维度矩阵法 (Dimensional Matrix) 针对复杂数据，定义多个正交（互不干扰）的分类维度。 1. 罗列属性：列出所有内容属性（如：价格、颜色、品牌、评分、新旧程度）。 2. 筛选高频：分析并确定用户最高频使用的筛选条件。 3. 设计逻辑：规划筛选器（Filter）和排序器（Sorter）的组合交互逻辑。 • 电商平台 • SaaS 后台系统 • 数据仪表盘 (构建矩阵结构)
实体关系建模 (Entity-Relationship Modeling) 从数据库逻辑反推产品结构，定义对象之间的关联网络。 1. 定义对象：确立核心实体（如：用户、订单、商品、评论）。 2. 定义关系：明确实体间关系（一对多、多对多、多对多等）。 3. 设计路径：基于关系设计跳转逻辑（例：商品↔评论，用户↔购买记录）。 • 构建自然结构（网状关联） • 社交网络产品 • 知识图谱/知识库

构建步骤总结

  1. 绘制草图 (Story Map / Site Map)：
    • 主流程用故事地图（线性）。
    • 导航体系用站点地图（层级）。
    • 搜索/筛选页用矩阵逻辑。
  2. **细化交互 (Flowchart)**：补充状态判断（如：未登录时点击购买会发生什么？）。
  3. **验证 (Tree Testing)**：找几个同事或用户跑通流程，修补漏洞。
  4. 输出文档：形成最终的 PRD（产品需求文档）中的架构图部分。

构建产品结构没有唯一的“标准答案”，最好的结构是最符合当前用户心智模型和业务目标的结构。通常会随着产品发展，从简单的线性/层级，逐渐演变为复杂的矩阵/自然混合结构。

原型及需求

产品设计流程中，框架层的设计内容，包括界面设计、导航设计和信息设计

  1. 概括待表达的信息，理清需求点

  2. 信息排序，需求点归类

  3. 组织界面语言——原型

工作顺序：需求采集–功能结构—原型设计—-需求文档（PRD）

PRD的作用：

  • 给谁看：交互设计师、视觉设计、项目经理、开发、测试、其他产品经理、其他需要了解业务逻辑的人
  • 作用：准确、直观、完整的表达产品需求；保证各角色沟通有依据；保证产品质量控制有标准；存档

PRD内容

文档组织按照三幕式结构，第一部分是需求背景、项目目标、修订记录，一般修订记录放到最前面，然后描述背景需求和项目目标，这一部分既重要又不重要，重要的是这是需求立足的根据和价值所在，说明了为什么要做这个产品，不重要是都要PRD文档怎么会不做，有些形式化。

接着第二部分是文档的核心：功能列表、逻辑展示和详细描述，功能列表可以使用表格展示，并标注优先级；逻辑展示是使用流程图描述产品的主要业务逻辑；详细描述就是描述每个功能的详细要求。

第三部分是数据分析和性能需求。

功能列表

注意：要拆分成最小的功能点；功能点之间相互独立；方便参与者理解需求，评估工作量

如何将功能拆分到最小？独立性和原子性

案例：网易云播放页面功能拆分

逻辑展示

弥补与程序员的差异；梳理思路；避免遗漏

业务流程：使用流程图展示产品逻辑，流程分为单角色和多角色，多角色的流程图就是泳道图。

页面流程：产品页面+路程图
