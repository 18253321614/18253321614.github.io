Agent 记忆

发表于 2026-03-26 分类于 产品经理 本文字数： 9.5k 阅读时长 ≈ 9 分钟

基本介绍

大语言模型LLM本身是无状态的，就像一条鱼只有七秒钟的记忆，每次对话结束就会忘掉一切。如果我想让LLM帮我写小说或者作为个人助理记住我的兴趣爱好，LLM无法做到，因为它记不住上下文。

Agent记忆就是给与大模型长期存储、检索和利用历史信息的能力，使其具备连续性和个性化。当前的Agent记忆实现的整体思路是将历史对话信息提出关键信息保存并进行管理，在与大模型对话时，根据用户的问题检索与问题有关的历史对话放到上下文中。

记忆分类记忆分为长期记忆和短期记忆。短期记忆又叫会话记忆，即在一次聊天中，大模型可以记住前面聊过的内容；长期记忆是指大模型能够记得用户通过不同上下文窗口与大模型从第一次到最后一次聊过的所有内容。

长期记忆是Agent记忆研究的重点，在认知心理学中，记忆时长在”1分钟—终生“这个时间段内的记忆叫做长期记忆，长期记忆又可以分为陈述性记忆和非陈述性记忆。

  • 陈述性：有意识的事实、事件和知识
    • 情景记忆：个人经历，例如去年去旅游
    • 语义记忆：客观事实，例如地球是圆的
  • 非陈述性：无意识、自动化的技能与习惯
    • 程序性记忆：工作技能，例如骑自行车
    • 关系记忆：人和人相处相成的信任和默契感等，具有弥散性
    • 其他

想要LLM更像人，就需要让大模型掌握人的长期记忆能力。

核心机制

以Mem0为例说明记忆存储、更新和检索

2025年4月，Mem0团队在arXiv上发布了论文《Mem0: Building Production-Ready AI Agents with Scalable Long-Term Memory》，提出了一种全新的以内存为中心的可扩展架构。

它并不只是简单地存储对话日志，而是通过 LLM 将非结构化对话转化为结构化的原子记忆（Atomic Memories），并通过智能冲突解决机制确保记忆的准确性和时效性。

  • 编排层 (Orchestration): Python SDK，负责流程控制。
  • 大语言模型 (LLM): 用于记忆提取、冲突判断、记忆重写（支持 OpenAI, Ollama, Anthropic 等）。
  • 嵌入模型 (Embedding Model): 将文本转化为向量（支持 text-embedding-3-small, nomic-embed-text 等）。
  • 向量数据库 (Vector Store): 存储记忆的向量表示，支持语义搜索（支持 Qdrant, Chroma, Milvus, PGVector, Redis）。
  • 图数据库 (可选，Mem0^G): 用于存储实体关系（支持 Neo4j 或内存图结构）。
  • 元数据存储: 通常与向量库共存，存储 user_id, timestamp, memory_id, score 等。

记忆存储

不是所有对话都存，需要过滤（Filtering）和摘要（Summarization）。

记忆提取

增量处理与上下文感知：Mem0不会等到对话结束才提取记忆，而是在每次交互后，都会触发记忆提取流程：

  • 首先获取当前对话片段，同时结合最近的记忆（user_id下的最近k条记忆）和历史消息序列，构建prompt，让大模型进行记忆提取，这样可以保证大模型提取记忆不会断章取义。

隐式划分

LLM进行记忆提取时，会被提示对不同信息进行标记，这对应了心理学中记忆分类的划分。例如事实、事件、技能、模式等。对于陈述性记忆，LLM会将这些信息转化为独立的、自由包含的字句，并去除代词，使其脱离语境也能被理解；而程序性记忆，则会提取规则性描述或元数据标签。

原子化

Mem0不存储长短落，而是存储小的记忆单元，例如用户说：“我平时喜欢吃辣，不过最近胃不太好，所以暂时吃清淡点，等好了再继续吃辣。”LLM会分析对话，输出json结构化记忆内容。

记忆更新

向量数据库（Vector DB，用于语义搜索）+ 关系型数据库（用于事实数据）。

当记忆信息进入系统，另一个挑战就是应该新增、更新还是舍弃。如果处理不当则会导致记忆混乱和矛盾。mem0采取可以种基于检索的决策机制来解决这个问题。

更新流程

从新对话中提取候选记忆合集，在向量数据库中进行相似度检索，找到已存在、语义相似的旧记忆

决策机制

LLM拿到新旧记忆后，会按照下列准则判断如何处理新记忆，输出动作（ADD、UPDATE、DELETE、NONE）：

  • 如果是新记忆，则直接新增
  • 如果新旧记忆主题相同但内容不同：融合生成一条新记忆，包含新状态和时间戳，并标记旧记忆为过时
  • 如果新记忆否定了旧记忆，则删除旧记忆
  • 如果是重复记忆则不做操作

防混乱

为了保证记忆清晰和逻辑一致，mem0采用以下策略：

  • 语义去重：语义高度相似只存储一条
  • 时间敏感处理：mem0的prompt强调时间状语的重要性。对于冲突信息会保留最新状态或者在记忆中明确时间范围
  • 原子更新而非直接覆盖：原子化存储记忆，更新居住地不会影响喜欢的食物
  • 置信度过滤：因为在记忆提取时会同时获取历史对话和记忆，所以可以判断用户是在开玩笑还是真的要记住

记忆检索

基于相似度的检索（RAG）、基于时间的检索、基于重要性的检索。

查询向量化

向量数据库的相似度查询

混合检索与重排

使用语义搜索+关键词搜索的方式获取相关记忆的原子片段，然后将两者搜索结果放一起进行重排序

组装上下文

首先对检索到的记忆进行加权分数计算，去其次去掉语义重复的信息，最后进行格式化输出，例如：”已知用户偏好：1. 喜欢冰美式 (更新于 2天前); 2. 对花生过敏 (更新于 1个月前)。”

图数据库

除了向量数据库之外，mem0还支持使用图数据库保存。图数据与关系数据库使用二维表格存储数据不一样，而是采用节点、关系和属性来描述存储数据，节点通过关系连接，例如小明是一个节点，美女是一个节点，通过一个关系进行连接，节点和关系都可以有属性来描述，每个节点都有一个id。在计算机中，就是一张有向图，通过有向边连接不同的节点来描述关系。图数据库适用于高度关联的数据，能够以极快的速度查询复杂的关系网络。Neo4j，是目前最流程的图数据库，同时也支持事务和查询。

图数据库的这种特性非常适合存储关系记忆，在mem0中也引入的图数据库，与向量数据库相互补充，更好的挖掘和保存记忆。基本工作模式是：

  • 向量数据库：负责模糊语义检索和快速筛查
  • 图数据库：负责精确关系推理、实体链接和结构化更新

写入流程

要求LLM不仅提取原子事实，还要识别实体 (Entities) 和 **关系 (Relations)**。在向量数据库中保存记忆信息时，两个数据库中的数据会双向关联，向量数据库中记忆片段增加图数据库中的节点id。

更新流程

根据用户输入，在向量数据库中查找到结构化的记忆片段，根据其中的节点id在图数据库中查找到相关节点，有大模型进行决策对两个数据库的相关数据进行增删改查

检索流程

首先在向量数据库中检索，然后在图数据库中检索，最后将向量检索到的“用户偏好上下文”和图检索到的“推理事实”合并。

代码示例

开源项目记忆架构设计

Openclaw

OpenClaw 的记忆系统分为三层：当前对话、长期记忆、每日记忆，每层有不同用途。

文件结构如下：

会话记忆记录当前对话的上下文，AI 能”记住”你们刚才聊了什么。保存到对话结束，可配置数天，储在~/.openclaw/sessions/

长期记忆是人工精选的长期记忆，相当于 AI 的”核心知识库”。永久保存，除非手动删除，存储在 ~/.openclaw/workspace/MEMORY.md

每日记忆是 AI 的”日记”，记录每天的原始信息。保留7-30天，存储在 ~/.openclaw/workspace/memory/

对于记忆文件，Openclaw在检索时支持使用向量检索与关键词检索的混合检索方式，如果要启动向量检索则需要配置嵌入模型。否则Openclaw知识定点读取或者词面匹配检索记忆文件。

OpenViking

OpenViking 是一个专为 AI Agent 设计的上下文数据库，用”文件系统范式”统一管理记忆、资源、技能，让 Agent 越用越聪明。类似Linux中思路，一切皆文件。所有的记忆、资源、技能被统一组织在一个虚拟文件系统中，每个条目通过 viking:// 协议拥有唯一的 URI。

OpenViking 借鉴了计算机图形学中 LOD（Level of Detail）的思想和 CPU 缓存层次的设计，在资源写入时就自动处理为三个层级：

  • L0（摘要 / Abstract）：约 100 个 Token。一句话概括内容，用于快速筛选相关性。相当于书名加一句话简介。
  • L1（概述 / Overview）：约 2000 个 Token。包含核心信息、结构要点和使用场景，供 Agent 在规划阶段决策。相当于书的目录和内容提要。
  • L2（详情 / Details）：完整原始数据，只在 Agent 确认需要深入阅读时才加载。就是完整正文。

在虚拟文件系统中，每个目录都有自己的 .abstract 和 .overview 文件：即目录的摘要和概述

举个例子。用户问”API 的认证方式有哪些？”

传统方案：把整个项目文档加载进来（几万甚至十几万 Token），让模型从中找答案。

OpenViking 方案：

  1. 读取项目根目录的 L0 摘要（~100 Token），确认包含 API 文档
  2. 读取 docs/ 目录的 L1 概述（~2000 Token），发现有 api/ 子目录
  3. 读取 api/ 目录的 L1 概述，发现 auth.md 与认证相关
  4. 只加载 auth.md 的 L2 全文

此外由于 OpenViking 的所有上下文都以统一格式存储、每个条目对应唯一 URI，整个检索过程天然可追溯。你可以清楚看到：

  • Agent 从哪个目录开始搜索
  • 递归进入了哪些子路径
  • 每个路径的相关性得分是多少
  • 为什么最终选择加载了特定资源
  • 整个检索链路如何一步步展开

每次会话结束时，开发者可以调用 session.commit() 触发记忆提取。系统异步分析任务执行结果和用户反馈，自动更新到 user/ 和 agent/ 的记忆目录。

这个过程覆盖两个维度：

  • 用户偏好记忆：用户多次表现出偏好简洁回答的风格，系统自动将偏好沉淀为记忆。后续回答自动贴合用户习惯。
  • 任务经验记忆：Agent 执行任务中积累的操作技巧、工具经验、解决策略，自动提取存储。下次遇到类似场景，直接调取”实战经验”辅助决策。

Openclaw可以使用open Viking或者mem0作为记忆插件
