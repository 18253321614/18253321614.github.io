RAG

发表于 2026-02-25 分类于 AI 本文字数： 5.9k 阅读时长 ≈ 5 分钟

传统RAG

背景：企业RAG比赛：https://github.com/trustbit/enterprise-rag-challenge/tree/main

基于公司年度报告构建一个问答系统，简单来说，比赛当天的流程如下：

  • 我们会收到来自随机挑选公司的100份年度报告，并需要在2.5小时内解析这些报告并构建一个数据这些报告是PDF格式，每份最长可达1000页。

  • 然后，系统会生成100个随机问题（基于预设模板），我们的RAG系统必须尽快回答这些问题。

所有问题都有确定的答案，例如：是/否；公司名称（某些情况下是多个公司名称）；领导职位头衔、推出的产品名称；数值指标：营收、商店数量等。

·每个答案都必须注明引用的页码，确保系统是从原文中得出答案，而不是输出虚假信息(hallucinate)。

解决方案

以下是冠军团队的方案

Ingestion

PDF解析成文本，如果有图片也是同样解析为文本信息

文本清洗、分块，降低信息密度，如果直接将一篇文档向量化之后存储，因为向量维度是固定的，那么对于这个向量来说信息密度就太高了。那么如何切片是一个问题，这需要根据文档的情况和问题的情况进行切分，例如300-500tokens、或者800tokens、1000tokens。宏观的问题就需要大一些，细节问题更多则需要小一些。因为tokens的大小代表的是信息密度不同。

将分块、清洗后的文本向量化后存入向量数据库，不同的是，此处使用了多个向量数据库来存储不同公司的财报数据，每个数据库使用公司名称作为Tag，这样在Retreive的时候效率会更高。100家公司用100个向量数据库。

Answering

所以用户提出一个Question后，通过路由选择，在全量数据库找到相关的数据库（表的概念）。然后将Question同样进行向量化，使用文本向量检索算法对该库进行Retrieval检索，检索到目标后，取出目标所在的父页面的所有内容，也就是ParentPage，然后进行reranking排序。

  • Retrieval检索（召回）：快速筛选潜在的chunk，速度快，例如从1000万个中选出1000个

  • Reranking排序（排序）：精确的排序，从1000个中选出10个。Rerank模型分为基于普通神经网络和基于LLM的

通过这两步获取到相关的上下文，RelevantContext。与此同时，获取到Question后，通过路由选择到适合该问题的提示词模板。

最终将RelevantContext和Question组装到PromptTemplate中，发给LLM，LLM给出结果。

基本概念

文本分块

在上述的Ingestion过程中简单介绍了文本切块，但是在实际应用中并没有这么简单。适当的切块方法可以对文本按照语义进行准确的划分，保证向量化的片段是一个意群，更有利于文本正确的召回，文本切换过长和过短都存在相应的问题：

  • 文本切块过长：
    • 优点：更好保留文档层面或段落层面的上下文信息
    • 缺点：文本片段向量化一般采用平均token或cls位置的token作为整个片段的向量化表示，提过片段太长会导致向量的语义太过稠密，进而损失更多细节的语义信息，从而导致召回准确率降低。同时过长的片段在交给LLM推理的时候也需要更多的推理成本，更大的模型上下文要求。
  • 文本切块过短：
    • 优点：更有利于保留文本中细粒度的语义信息
    • 缺点：容易丢失段落和文章层面的主题信息和跨段落的上下文信息，同时过于琐碎的文本块也不利于大模型总结。

整体来说文本切块没有一个最佳的实践，而是需要根据任务类型和资料的情况确定，例如像说明书这样的文本就可以切的小一些，而论文这种逻辑紧凑的资料也需要切块的大一些。

常用的一些文本切块方法有固定大小文本切块、特殊格式文本切块、NLTK、基于深度学些的文本切块，文本切块的一个核心是找到一个切点，即文本切块的边界，特殊格式文本切块、NLTK、基于深度学些的文本切块都是确定一个边界，基于深度学习的文本切块方法基本思想是：将两个连续的句子交给BERT，让其预测第二个句子是第一句子下一句的概率，以此拉确定切分的边界。

文本行量化

将一段文本使用一个多维向量表示。例如使用一个768维的向量表示一句话，例如“我喜欢你”这句话使用某个模型将其一个向量[11,23,12,34,534……]表示。这个模型叫做嵌入模型EmbeddingModel。我们生活的空间是三维空间，使用一个三维向量就可以表示空间中的任何一个点。而这个768维的向量表示的是一个高维空间。不同的语句、片段等经过嵌入模型都会转化为这个高维空间的一个点，衡量嵌入模型EmbeddingModel优劣的一个标准是：均匀性和对齐性。即首先语义相似的文本片段在高维空间中的距离应该比较近，同时在整个空间中各文本片段应该是均匀分布，不能过度集中。

EmbeddingModel是一个RAG系统是否表现优良的关键，因为在Ingestion和Answering两个阶段需要用到。

  • 将包含知识的文本片段行量化并存储到向量数据库中
  • 将用户查询文本实时向量化，以便从数据库中检索响应的文本

只有EmbeddingModel足够懂你的资料和提的问题，才能够召回和问题足够相关的资料，让LLM根据资料进行回答，降低产生幻觉的风险。

Embedding模型有很多，在https://huggingface.co/spaces/mteb/leaderboard中可以看到相关评测结果，但是在很多时候，尤其在垂直领域或者对检索精度非常高的场景下，通用的Embedding模型往往表现不够理想，例如一些垂直领域，通用模型无法准确把握专业术语之间的相关性，以智能客服为例，企业内部会有一些特有的表达方式和业务逻辑，通用的Embedding Model难以理解，此时就需要对Embedding Model进行微调。如何判断是否需要微调？至于如何微调后面的文章在讲

Retrieval检索

在Retrieval或者说召回阶段使用的文本向量检索模型有两种：

  • 基于BERT、GPT等深度学习模型的稠密向量检索算法。该算法是根据语义相似度进行检索，例如BGE、M3E、GTE等
  • 以TF-IDF、BM25为代表的稀疏向量检索模型。该算法是基于文本相似度进行检索

因此在召回阶段可以使用混合检索的方式来召回更精准的相似片段。此外针对用户的提问，可以使用LLM进行改写，例如一些中英文互译、专业术语优化等，帮助更加全面的召回相关片段，基本步骤如下：

  1. 获取用户检索的问题，并改写用户问题
  2. 进行第一次数据库检索：使用关键词检索算法在向量数据库中进行检索，例如bm25进行关键词检索，获取相关片段doc1
  3. 进行第二次数据库检索：对改写后的用户问题进行向量化，基于向量相似度检索算法获取相关片段doc2
  4. 对两次检索结果进行合并，合并之后进行重排序，重排序的结果放到提示词中发送给LLM

此外在检索有两种类型：

  • 对称检索：搜索与检索问题文本相似的片段
  • 非对称检索：搜索与检索问题答案文本相似的片片段

明显在RAG场景下，应该采用非对称检索，所幸当下的检索算法中，不管是稠密检索还是稀疏检索算法都支持非对称检索

重排序

重排序是把召回的文本片段和用户问题在做一次相似度检验，只保留相似度高的片段，以此来降低模型推理成本和幻觉产生的概率。主流的排序模型包括：

  • 基于交叉编码器 (Cross-Encoder) 的模型：这是目前效果最好、精度最高的重排序模型
  • 基于多向量/后期交互的模型：在效果和成本之间取得了很好的平衡，速度也不错
  • 基于大语言模型 (LLM) 的排序：利用大模型强大的语义理解和推理能力进行排序，泛化能力好但成本较高、速度慢
  • 轻量级/高效模型：模型体积小，推理速度快，适合实时性要求高的应用，但精度相对有所牺牲。

召回环节优化

针对用户问题，从向量数据库中检索与之相似或者与之答案相似的文本一同发给LLM得出回答，召回内容的质量直接影响模型回复的质量，因此在召回环节如何让RAG系统获得相关文本非常重要，因此也产生了很多优化方法。上述冠军方案中所使用的分库存储不同公司的财报、重排序、Parent Page就是其中之三，重排序已经上面已经写过

分库存储是使用的元数据优化方法，即在数据库上打上标签，例如使用数据库名字，在文本召回时首先路由到对应的向量数据库，在进行进一步检索，就可以提升文本召回的效率和准确度。ParentPage是将在文本切块为较小的片段，保证向量化之后拥有更细粒度的语义信息，在根据用户问题检索到相关片段，不使用检索到的片段作为模型回复的上下文信息，而是使用检索到的片段的父文本。

此外还有

  • 短文本信息增强：为分块后的短文本增加全局信息，例如段落的前三句（总分结构）、长文本标题、长文本主旨句（可以有LLM总结），一起加到短文本上，再进行向量化
  • 召回内容的上下文扩充：ParentPage就是这种方法，此外也可以选择检索到的短文本片段前后的n个片段共同作为LLM回复的上下文
  • 多向量表示：与短本增强相似，但是区别是全局信息、假设性问题等单独向量化，作为补充向量，然后将补充向量与短文本向量建立映射关系，检索到相关补充向量后在提取短文本向量发给LLM

上面提到的召回优化都是针对数据库中的内容，另一个优化方向是用户查询，用户输入的问题往往不是一个很好的提示词，所以在进行数据库检索时，可以对用户的查询内容进行优化，例如进行扩充，或者由大模型直接回复生成“假答案”，然后将假答案和原问题放在一起作为融合查询条件进行查询。

Agentic RAG

Agentic RAG 描述了一种基于 AI Agent 的 RAG 实现。具体来说，它将 AI Agent 整合到 RAG 流程中，让Agent在整个信息检索和生成过程中主动思考和决策，而不是仅仅是简单的接受用户查询—检索文档数据库—基于文档生成答案，以克服传统 RAG 的局限性。

过程解析

接受用户查询

获取用户查询后，首先对原始的用户查询进行重写，转化为更精准可检索的形式，然后分析是否需要获取额外的信息，例如“最近天气对公司销售有什么影响”，此时LLM判断就要获取公司销售数据（已持久化存储到向量数据库）和天气信息（调用MCP工具），因此进入第二条线，到数据库中查询销售数据并获取最近的天气数据，进而由LLM生成答案。

多元数据检索

在DataSource中，整合多源数据，包括内部知识如内部资料和业务系统的数据库，和外部数据如互联网中的数据，例如在客户支持的场景中，系统可以查看用户历史工单（数据库查询）、检索产品使用手册（向量数据库）、最新的技术更新信息，进而整合多源数据生成回复。

答案分析

在生成结果后，并不直接回复给用户，而是评估用户查询和生成回复的相关性和准确性，如果回复不满足要求则调整检索策略进行重试或进一步对答案进行修改

自我修正

传统系统遇到无法回答的问题往往会直接告诉用户”无法解决”，Agentic RAG则会识别信息鸿沟，主动寻找补充信息源。

主流方案

Nvidia Agentic RAG：https://github.com/NVIDIA/workbench-example-agentic-rag/blob/main/code/langgraph_rag_agent_llama3_nvidia_nim.ipynb

这里以Nvidia Agentic RAG为例介绍，整理思路是获取用户提问后，进入Routing（例如上述传统RAG中利用Tag进行路由）

  • 如果知识库中有相关的资料（related index）则Retieve检索文档，然后对检索到的文档片段进行重排序（grade），进一步判断文档片段是否与用户问题相关
    • 如果相关则交给LLM生成回复，生成回复后判断是否存在幻觉
      • 如果存在幻觉，则重新生成
      • 没有幻觉则判断是否符合用户问题
        • 如果与用户问题相符则返回最终answer
        • 不相符则直接进行网络搜索
    • 如果不相关则进行网络查询
  • 若知识库中没有相关资料则直接进行网络查询，将网络检索到的信息交给大模型生成回复
