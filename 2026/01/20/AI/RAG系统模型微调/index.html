RAG系统模型微调

RAG系统的模型

在一个RAG系统中，所用可能到的模型类型如下图所示，每个模型都代表了RAG系统可优化的点，当然并不是所有的RAG系统都采用如此多的模型来约束和优化系统行为，本身还是一个成本和效果以及用户需求的平衡。

在离线准备阶段

文档解析与版面分析模型：需要将将非结构化文档（PDF, PPT, 扫描件）高质量转化为文本/Markdown。使用到文档解析与版面分析模型，例如LayoutLMv3 / Donut，根据文档类型选择不同的模型。

嵌入模型（Embedding）：得到文本/Markdown后，需要使用嵌入模型讲文本转化为向量存储到向量数据库

在线问答阶段：

查询改写/扩展模型：获取用户查询后对用户查询内容进行改写/扩展。例如用户问“它所少钱？”“它”就要改写。可以使用一些轻量的LLM例如Qwen-1.5B这样的小参数量的LLM。

意图识别模型：获取用户查询后，分析用户意图，例如用户只是闲聊就没有必要查询知识库、敏感问题直接拦截、数学计算/代码执行等调用工具，事实查询时才查询知识库。有一些轻量级的分类模型可以胜任。

问题拆解模型：将复杂的问题拆解为多个子问题，通常使用LLM通过Prompt拆解

重排序模型（Rerank）：根据用户问题检索知识片段后，使用重排序模型对知识片段与用户问题的相关性进行评价，舍弃相关性低的文本片段，降低推理成本和风险概率。

主LLM：根据资料回答用户问题，可以使用deepseek r1、google Gemini等

幻觉检测：在输出给用户前，二次检查生成的答案是否忠实于检索到的上下文。

安全护栏模型： 过滤输入中的攻击提示（Prompt Injection）和输出中的有害内容。例如navdia的NeMo Guardrails。

| 模型类型 | 必要性 | 适用场景 | 替代方案 |
| :— | :— | :— | :|
| LLM + Embedding + Rerank | ⭐⭐⭐ 必须 | 所有 RAG 系统 | 无 |
| 文档解析 (Layout/OCR) | ⭐⭐⭐ 必须 (若有PDF/图片) | 含扫描件、复杂排版的知识库 | 简单的文本提取器 (效果差) |
| 查询改写 (Rewriter) | ⭐⭐ 推荐 | 多轮对话、指代消解场景 | 用主 LLM 兼任 (增加延迟) |
| 意图识别 (Router) | ⭐⭐ 推荐 | 需节省成本、区分任务类型 | 用主 LLM 兼任 |
| 幻觉检测 (Verifier) | ⭐ 可选 (高价值场景必选) | 医疗、法律、金融报告 | 人工审核、Prompt 约束 |
| 安全护栏 (Guardrails) | ⭐⭐ 推荐 (公网服务必选) | 面向公众开放的服务 | 简单的关键词过滤 |

模型微调

在垂直领域（如医疗、法律、金融、工业制造等）构建 RAG 系统时，“通用模型直接可用”往往是一个误区。垂直领域的术语、逻辑和表达习惯与通用语料差异巨大。

必须微调

嵌入模型和重排序模型是必须微调的

嵌入模型微调可以极大提高召回率，因为对于通用的嵌入模型对专业领域的语义理解不够准确和丰富，例如苹果在科技新闻中是公司，在农业新闻中是水果；心梗和心肌梗塞在向量空间应该离得非常近，除了这种术语漂移的问题，还存在同义词缺失的问题，例如法律中的甲方和发包方、编程中的debug和defect；最后是一些硬负例，即字面相似但是含义相反的情况，例如”逆变器报绝缘阻抗低怎么处理？”和”《逆变器故障代码手册》第 5 章：绝缘阻抗高的原因分析…”。这一些都需要在微调时告诉模型，在文本向量化时才能更好的理解语义。

重排序模型的作用是进一步筛选检索到的文本片段，垂直领域的一些细微差别需要模型具备较强的识别能力，因为重排序模型本来做的就是精细活。例如通用模型可能觉得包含更多关键字的是好片段，而垂直领域可能认为包含特定否定词的更好

建议微调

查询扩展/扩写模型和意图识别模型，微调后能显著提升用户体现和系统效率，特别是处理复杂查询时。

查询改写/扩展模型可以将用户口语化的提问转写为符合知识库文档风格的专业查询语句，此外还可以帮助LLM是被一些行业黑话与缩写，例如输入心梗，改写为急性心肌梗死诊疗指南。在多轮对话中需要结合上下文进行改写，例如上文用户询问了关于openclaw的问题，下一个问题用户用“它”指代，这时候也需要模型去进行指代消解。

意图识别模型可以完成业务的精准路由，对完意识识别也可以使用通用轻量的LLM，但使用经过微调后的模型反应更快、更准确

视情况而定

文档解析模型、幻觉生成检测模型、LLM

这单独立聊一聊为什么LLM微调的优先级比较低？因为RAG系统的核心就是外挂知识，理论上不需要微调LLM来记忆知识。但是在一些特殊场景下：

  1. 风格统一：例如要求输出严格的公文格式、法律文书、特殊口吻等
  2. 指令遵循：让模型更好的遵循复杂领域约束，例如必须引用法律条文
  3. 领域思维链：让模型学会领域特有的推理步骤

注意：在 RAG 架构中，对 LLM 进行指令微调（SFT, Supervised Fine-Tuning）不再是用来“注入知识”的，而是用来“规范行为”的。

微调方案

对于模型微调来说，调整的其实只有两个方面一个是模型的参数，另一个是模型的架构，而模型的架构一般情况不需要我们改。所以调整的是参数，也就是权重矩阵，因此训练数据集的构建是核心。

数据集构建

数据采集

对于光伏场景，数据主要来源于两部分：

  1. 运维侧：设备说明书、历史故障工单、巡检日志。
  2. 预测微调侧：气象特征描述、历史相似日预测偏差复盘报告、人工修正规则库。
数据集构建

构建光伏领域的 (Query, Positive, Hard Negative) 三元组数据。例如：

这里解释一下正例 (Positive)、负例（Negative）和硬负例（Hard Negative）

正例：与用户查询（Query）语义高度相关，能够直接回答用户问题或提供关键信息的文档片段。

负例/软负例：与查询完全无关，既没有语义关联，也没有关键词重叠的文档。

硬负例：字面上非常相似（包含大量相同关键词），但语义上不相关，或者答案错误/不匹配的文档。

数据集的构建方法：

  1. 数据对齐与正样本生成：例如可以利用通用大模型如Deepseek等，输入一段光伏预测规则或者运维手册的chunk，让大模型生成响应的query，这样就可以得到一堆query和pos
  2. 硬负例挖掘：训练数据集是一个三元组，除了query、pos还有neg，neg中分为负例和硬负例，尤其是硬负例的学习，这是提升模型效果的核心，简单的随机负例样本无法让模型学到细微差别，需要利用字面相似但是语义不同的文本作为硬负例样本。如上示例中：第一个负样本字面高度相似（天气描述完全一致），但讲的是运维故障而非预测修正；第二个负样本是预测修正，但季节和逻辑完全相反。

代码实现

针对BGE-M3有两种微调方法：

  1. 使用 BGE 官方推荐的 FlagEmbedding 库，它完美支持 BGE-M3 的微调
  2. 原生 PyTorch/HuggingFace，需要开发者手写 Loss、DataLoader 和训练 Loop

在分布式光伏这种既有专业术语（如 PID效应）、又有数值规律（如 11:00-13:00 功率波谷）的复杂场景下需要结合稠密检索、稀疏检索，因此更适合使用官方推荐的 FlagEmbedding 库。

重排序模型微调

BGE-Reranker-v2（目前中文效果最好的开源重排序模型之一）的微调

重排序模型微调和嵌入模型微调的区别

特性 Embedding 微调 Reranker 微调
输入格式 单独编码 Query 和 Doc，再算相似度 拼接输入 [CLS] Query [SEP] Doc [SEP]
输出目标 向量空间距离 (Contrastive Loss) 二分类概率 (Cross-Entropy Loss) (判断该 Doc 是否与 Query 相关)
计算方式 双塔结构 (Dual Encoder)，速度快 交叉编码器 (Cross Encoder)，速度慢但精度高
数据需求 需要大量负例构建对比 需要明确的 (Query, Positive, Hard Negative) 三元组
光伏场景价值 提升召回率 (Recall) 提升准确率 (Precision) 和 NDCG
数据集构建

与 Embedding 的三元组不同，Rerank 模型微调需要的是 **(Query, Passage, Label)**，其中 Label 通常是 0 或 1（或者相关度得分）。

  • **正样本 (Score 1)**：Query 与对应的正确修正规则/运维手册。
  • 负样本 (Score 0)：极其关键。不能用随机文档，必须使用 Embedding 模型召回的 Top 2-10 的非目标文档。
    • 业务逻辑：既然 Embedding 已经把这些文档排在前面了，说明它们非常有迷惑性。Rerank 的任务就是学习如何把这些“高仿”文档踢出去。
